Working With AI Agents

LiteSkill VR's MCP server is a read-write interface: agents don't just dump findings, they query existing research, review what past sessions discovered, build on prior analysis, and avoid duplicating work. This page covers connecting an agent and running a productive session. For the full tool list, see MCP tools.

Connect

The embedded server runs over HTTP on 127.0.0.1 and starts when you open a project; the port shows in the status bar. Configure Claude Code:

{
          "mcpServers": {
            "liteskill": {
              "url": "http://127.0.0.1:27182"
            }
          }
        }

Running headless or want the agent to own the process lifecycle? Use the stdio transport instead.

Author identity

Every write is attributed automatically — you never pass an author per call. Over HTTP, the agent identifies itself with a header on each request:

X-LiteSkill-Author: claude-code

Everything that connection creates is stamped with that author and author_type: "agent". With no header, writes default to anonymous-agent; over stdio they are attributed to stdio-agent. UI edits you make are stamped human with your OS username. Consistent author identity is what makes changes_since and bulk_delete precise — so give each agent a distinct, stable author name.

Pair it with an analysis backend

LiteSkill VR documents; it does not analyze. Connect an analysis MCP server in the same agent so it can do the work and record it in one loop — for example pyghidra-mcp for Ghidra decompilation and xrefs. The agent reads from Ghidra and writes findings into LiteSkill VR.

A productive session

  1. Orient — project_summary, tag_list, connection_type_list establish current state and the registered vocabulary.
  2. Catch up — on a return visit, changes_since shows everything new since last time.
  3. Review a target — item_get returns everything known about an item before the agent adds more.
  4. Document in batches — ioi_create_batch and note_create_batch write atomically; if one entry is invalid the whole batch is rejected with a specific, fixable error.
  5. Connect — connection_create_batch records how items and findings relate.
  6. Query — search for full-text, filter for structured (e.g. all critical findings).
  7. Recover — bulk_delete undoes a bad session in one call (it requires at least one filter, so it can't wipe everything by accident).

Why it stays clean

  • Registered vocabularies — tags and connection types must exist before use, so you never end up with buffer-overflow vs memory-corruption vs overflow across sessions.
  • Duplicate detection — ioi_create returns a duplicate_warning when a similar finding already exists on the same item (the create still succeeds; the agent decides).
  • Read before write — the orient/catch-up steps mean agents build on prior work instead of re-discovering it.

Tips

  • Keep LiteSkill VR open while the agent works — findings stream into the UI in real time over IPC.
  • Multiple agents can connect at once (e.g. Claude Code and Codex); distinct author names keep their contributions separable.
  • Register project-specific tags and connection types up front so agents have the vocabulary they need.
