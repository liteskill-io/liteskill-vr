Architecture

LiteSkill VR is a Tauri v2 application: a React/TypeScript frontend over a Rust backend that owns the SQLite project store and hosts the MCP server.

Components

Frontend (TypeScript / React)

Human/agent parity. The frontend is a full read-write client — every mutating MCP tool has a human CRUD affordance (human ≥ agent), enforced in CI. See The interface.

  • Dashboard — project overview: items, severity/triage breakdown, recent findings
  • Tab bar — open items as tabs (open/close/switch); each tab is an item
  • Item detail — an item's notes, items of interest, connections, with create/edit/delete
  • Connection map — Cytoscape.js graph of items and connections; click a node to open it
  • Sidebar — navigation (items / by severity / explanations / managers)
  • Tag & Connection-Type managers, Search/Filter view — parity with the vocab + search/filter tools
  • Zustand store — client view state; holds the latest project snapshot

Rust backend

  • Project store — CRUD + delete for all entities; SQLite (one .lsvr file per project) with FTS5 for full-text search.
  • MCP server — streamable-HTTP on 127.0.0.1 (the /mcp endpoint; also a stdio transport in the headless binary), started automatically when a project is opened.

The Rust crate exposes db and mcp as pure modules — nothing in them depends on Tauri — so both the desktop app and the standalone headless binary link against the same code path.

IPC contract

Deliberately tiny: one read command, one write command, one event.

invoke("project_snapshot") → { items, details, tags,   // READ
  connection_types, explanations, explanation_details, mcp_port }

invoke("mcp_call", { tool, args }) → result            // WRITE (any tool call)
  // Runs the SAME handlers::dispatch the MCP server uses, stamped
  // author_type = "human" with the OS username — this is what makes
  // human and agent writes provably identical (parity).

listen("db-changed", callback)  // emitted on every backend mutation; the
                        // frontend responds by re-fetching the snapshot

One write path (dispatch) and one coarse sync signal (db-changed) — the same code for UI and agent writes.

Persistence

Each project is a single .lsvr file (SQLite, WAL mode, foreign keys on). Projects can be backed up or shared by copying the file.

Implementation status. The desktop app currently opens (or creates) project.lsvr in its working directory; an open/new file dialog and OS file-association are planned (see File formats).
