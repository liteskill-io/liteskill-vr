Architecture

LiteSkill VR is a Tauri v2 application: a React/TypeScript frontend over a Rust backend that owns the SQLite project store and hosts the MCP server.

┌─────────────────────────────────────────────────┐
        │               LiteSkill VR (Tauri)               │
        │  ┌───────────────────────────────────────────┐  │
        │  │           Frontend (React/TS)             │  │
        │  │  Tabbed Views · Search · Connection Map   │  │
        │  │  Item Browser · Notes · Tags              │  │
        │  └─────────────────┬─────────────────────────┘  │
        │                    │ IPC                         │
        │  ┌─────────────────┴─────────────────────────┐  │
        │  │           Rust Backend                    │  │
        │  │  ┌─────────────┐    ┌─────────────────┐   │  │
        │  │  │ Project     │    │ MCP Server      │   │  │
        │  │  │ Store       │◄──►│ (localhost)     │   │  │
        │  │  │ SQLite      │    │ items, notes,   │   │  │
        │  │  │ (.lsvr)     │    │ ioi, conns…     │   │  │
        │  │  └─────────────┘    └─────────────────┘   │  │
        │  └───────────────────────────────────────────┘  │
        └─────────────────────────────────────────────────┘
               ▲                        ▲
               │ UI interaction         │ MCP (HTTP+SSE on 127.0.0.1)

Components

Frontend (TypeScript / React)

  • Tab bar — open items as tabs, browser-like navigation
  • Item browser — all items with status and summary stats
  • Item detail — notes, items of interest, connections
  • Connection map — Cytoscape.js graph of items and connections
  • Search — full-text search across all entity types
  • Tag manager — registered tags and connection types
  • Zustand store — client state, synced via Tauri IPC

Rust backend

  • Project store — CRUD + delete for all entities; SQLite (one .lsvr file per project) with FTS5 for full-text search.
  • MCP server — HTTP+SSE on 127.0.0.1, started automatically when a project is opened.

The Rust crate exposes db and mcp as pure modules — nothing in them depends on Tauri — so both the desktop app and the standalone headless binary link against the same code path.

IPC contract

Frontend ↔ backend communication uses Tauri invoke (request/response) and listen/emit (events).

invoke("project_get") → Project
        invoke("item_list", { filters }) → Item[]
        invoke("item_get", { id }) → ItemDetail (item + notes + ioi + connections)
        invoke("item_create", { data }) → Item
        invoke("item_update", { id, data }) → Item
        invoke("item_delete", { id }) → void
        invoke("note_create", { item_id, data }) → Note
        invoke("ioi_create", { item_id, data }) → ItemOfInterest
        invoke("connection_create", { data }) → Connection
        invoke("connection_list", { entity_id }) → Connection[] (both directions)
        invoke("connection_list_all") → Connection[] (project-wide)
        invoke("tag_list") → Tag[]
        invoke("connection_type_list") → ConnectionType[]
        invoke("search", { query, filters }) → SearchResult[]
        invoke("changes_since", { timestamp }) → ChangeSet

        listen("entity_changed", callback) // fired when MCP or UI mutates data

Persistence

Each project is a single .lsvr file (SQLite with a custom extension). The app opens project files via a standard file dialog. Projects can be backed up or shared by copying the file.
