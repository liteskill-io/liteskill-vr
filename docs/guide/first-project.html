Your First Project

This walkthrough takes you from a fresh install to your first documented finding — with an AI agent helping. It assumes LiteSkill VR is already installed.

Note. Today the desktop UI is a read-only viewer — you don't type findings into it. Items, notes, and findings are written by an AI agent over MCP, and the UI shows them live. This walkthrough reflects that flow.

1. Open the project

Launch LiteSkill VR. It opens (or creates) project.lsvr in its working directory — this single SQLite file is your project, so keep it somewhere you back up. A new project comes pre-loaded with the default tags and connection types, so an agent can tag findings immediately.

The status bar confirms the MCP server is listening and shows its port. Keep that port handy — your agent connects to it next.

2. What an item is

An item is anything you're analyzing — a binary, a shared object, a kernel module, a script, a config file. It is a documentation entry, not the file itself: you (well, your agent) records metadata such as name (e.g. httpd), type (e.g. elf), optional path and architecture, and registered tags — nothing is uploaded. Items are created through MCP (step 4), not typed into the UI.

3. Connect an AI agent

Point Claude Code (or Codex) at the MCP server shown in the status bar:

{
          "mcpServers": {
            "liteskill": {
              "url": "http://127.0.0.1:27182"
            }
          }
        }

Use the port from your status bar. For per-agent attribution, set the author header (see Working with AI agents). Ask the agent to orient itself — it calls project_summary — then to create your first item (e.g. the httpd binary) with item_create. It appears in the UI immediately.

4. Let the agent document

With an analysis backend such as pyghidra-mcp connected too, ask your agent to analyze the binary and record findings. It writes items of interest, notes, and connections through MCP — and they appear in your UI live, no refresh needed. A typical agent turn batches its writes so they all land atomically.

5. Review and connect the dots

Browse the Dashboard and open items as tabs to read what the agent recorded, and switch to the Connection map to see how items relate across the project — which binary links which library, which script spawns which daemon. To triage or correct, drive the agent: ask it to filter for all critical items of interest, to fix or ioi_delete a false positive, or — if a whole session went sideways — to undo its own batch with bulk_delete. (UI-side editing is planned; today corrections go through the agent over MCP.)

6. Share or archive

Your project is one .lsvr file. Copy it to back it up, hand it to a teammate, or archive it at the end of an engagement. Deliverable exports (markdown report, JSON, SARIF) are planned.

Next session? Your agent calls changes_since with the timestamp of its last run to catch up on everything that changed before continuing.
