Your First Project

This walkthrough takes you from a fresh install to your first documented finding — with an AI agent helping. It assumes LiteSkill VR is already installed.

1. Create a project

Launch LiteSkill VR and create a new project. You choose where the .lsvr file lives — this single SQLite file is your project, so put it somewhere you back up. A new project comes pre-loaded with the default tags and connection types, so you can start tagging findings immediately.

The status bar confirms the MCP server is listening and shows its port. Keep that port handy — your agent connects to it in step 4.

2. Add items

An item is anything you're analyzing — a binary, a shared object, a kernel module, a script, a config file. It is a documentation entry, not the file itself, so you record metadata about it rather than uploading anything.

Add an item from the project overview and fill in what you know:

  • Name — e.g. httpd
  • Type — freeform, e.g. elf, shared_object, config
  • Path / architecture — optional, for your own reference (e.g. /usr/bin/httpd, arm32)
  • Tags — drawn from the registered set

3. Document a finding by hand

Open the item as a tab. Press n to add an item of interest — a function, a string, a config line, a suspicious pattern, a potential bug. Give it a title, a description (markdown), an optional location (address/line/symbol), and a severity. Press a to attach a free-form note. This is the same data your agent will write — you're just doing it manually first to see the shape of it.

4. Connect an AI agent

Point Claude Code (or Codex) at the MCP server shown in the status bar:

{
          "mcpServers": {
            "liteskill": {
              "url": "http://127.0.0.1:27182"
            }
          }
        }

Use the port from your status bar. For per-agent attribution, set the author header (see Working with AI agents). Now ask your agent to orient itself — it will call project_summary and see the item you created.

5. Let the agent document

With an analysis backend such as pyghidra-mcp connected too, ask your agent to analyze the binary and record findings. It writes items of interest, notes, and connections through MCP — and they appear in your UI live, no refresh needed. A typical agent turn batches its writes so they all land atomically.

6. Review and connect the dots

Switch to the Connection map to see how items relate across the project — which binary links which library, which script spawns which daemon. Use ⌘K/Ctrl+K to search everything, and the filter view (e.g. all critical items of interest) to triage. Edit or delete anything the agent got wrong; if a whole session went sideways, an agent can undo its own batch with bulk_delete.

7. Share or archive

Your project is one .lsvr file. Copy it to back it up, hand it to a teammate, or archive it at the end of an engagement. Export a markdown report, JSON, or SARIF when you need a deliverable.

Next session? Your agent calls changes_since with the timestamp of its last run to catch up on everything that changed — including your manual edits — before continuing.
